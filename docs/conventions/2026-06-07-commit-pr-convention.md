# 커밋 & PR 컨벤션

이 문서는 본 저장소의 커밋 메시지와 Pull Request 작성 규칙의 단일 진실 원천(SoT)이다. 운영용 템플릿(`.gitmessage`, `.github/pull_request_template.md`)은 이 문서에서 파생되며, 둘이 어긋나면 이 문서가 우선한다.

본 컨벤션은 1인 프로젝트를 전제로 하되, 제3자가 커밋·PR 이력만으로 의사결정 과정을 따라 읽을 수 있도록 설계되었다.

## 1. 커밋 메시지

### 1.1. 형식

커밋 메시지는 제목·본문·`Refs:`·트레일러 네 부분으로 구성한다.

```
<type>(<scope>): 제목

본문 ("왜" 중심, 72자 wrap)

Refs: <plan/spec/task/PR/runbook 링크>
Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
```

- 언어: `type`·`scope`는 영어, 제목·본문은 한국어.
- 제목은 50자 이하, 마침표 없음.
- 본문은 72자 기준으로 줄바꿈하고, *무엇을* 바꿨는지(diff가 보여주는 것)가 아니라 *왜* 바꿨는지를 적는다.
- 본문 구성은 §2.2 가독성 원칙을 따른다 — 도입 문장(맥락·왜)을 먼저 쓰고, 세부·변경 항목은 명사형 불릿으로 나열한다.
- 개별 커밋 본문은 그 커밋이 직접 해결한 이슈를 5~15줄로 적는다. 큰 트러블슈팅은 §3의 기록 위치를 따른다.

### 1.2. type / scope

`type`은 다음 10종만 사용한다.

- `feat` `fix` `chore` `docs` `refactor` `test` `perf` `ci` `build` `revert`

`scope`는 다음 세트에서 고른다.

- `infra` `adapters` `ingest` `pipeline` `storage` `enrich` `dbt` `serve` `slo` `airflow` `env`
- `docs` 커밋에 한해 토픽 scope(예: `spec`, `plan`, `convention`)를 허용한다.

### 1.3. 트레일러

모든 커밋 본문 끝에 아래 트레일러를 붙인다.

```
Co-Authored-By: Claude Opus <버전> (1M context) <noreply@anthropic.com>
```

- `<버전>`은 실제 작업한 모델을 반영한다. 4.8로 작업하면 `4.8`, 과거 4.7로 작업한 커밋은 `4.7`.
- 이 트레일러는 커밋 단위로 AI 활용을 투명하게 남기기 위한 것이며, PR에는 적용하지 않는다(§2.7 참조).

## 2. Pull Request

### 2.1. PR 본문 구조

PR 본문은 아래 7개 섹션으로 구성한다. Squash merge를 쓰므로(§2.5) 이 PR 본문이 곧 main에 남는 단일 커밋의 본문이 된다.

각 헤더 아래는 먼저 소제목 성격의 한 문장으로 맥락을 제시하고, 그다음 불릿으로 세부를 명사형으로 전개한다(§2.2).

섹션 1~7은 세부 항목이 여럿이면 `###`(필요 시 `####` 이하) 소제목으로 항목을 나눠 읽기 쉽게 한다(번호 규칙은 §2.2).

#### 1. 개요

PR의 맥락·목적·결과·기대 효과를 위에서 아래로 자기완결적으로 요약한다. 세부 변경·검증은 뒤의 3·4 섹션이 펼치고, 1.3은 그 핵심만 추린다.

- `### 1.1. 배경` — 어떤 단계(plan의 Task)인지 + 직전 단계의 부족함·이 PR이 필요한 이유.
- `### 1.2. 목적` — 이 PR이 무엇을 하는지(한 줄 목표) + 이후 단계가 무엇을 전제로 시작 가능한지.
- `### 1.3. 결과` — 실제로 무엇을 해결했는지 핵심만. 항목이 여럿이면 `#### 1.3.1. 해결한 문제` · `#### 1.3.2. 변경 요약` · `#### 1.3.3. 검증`처럼 세부 소제목으로 나눈다.
- `### 1.4. 기대 효과` — 머지 후 변화 + SLO / 비용 / 운영 측면 영향. 불릿이 아닌 별도 sub-헤더로 분리해 도드라지게 한다.

#### 2. 의사결정 & Trade-off

의사결정 / 대안 검토 / 채택 사유 / 원안에서 달라진 점 표 / review 라운드의 Important fix를 적는다. diff가 그대로 보여주는 자명한 코드 narrative는 제외한다. 큰 트러블슈팅은 `docs/troubleshooting/<주제>.md`에 별도 문서로 archive하고 링크 + 한 줄 요약만 남긴다.

#### 3. 변경 사항

파일 단위 narrative. *무엇이* 바뀌었는지 불릿으로 적는다.

#### 4. 검증

명령어 + 결과 발췌. 명령어 옆 `#` 주석은 한국어로 단다. 스크린샷 / 메시지 샘플 / row count / SLO 측정값 등 실제 실행 증거를 포함한다.

#### 5. 장애 시나리오 & 롤백 전략

머지 후 잘못되면 어떤 형태로 잘못될 수 있는지 + 롤백 방법을 적는다. 데이터 손실 / 멱등성 / schema / SLO / 비용 / 보안 / 계층(streaming · cron · batch ops) 의존 중 해당하는 것만 다룬다. 해당이 없으면 "잠재 위험 없음. `git revert`로 롤백 가능." 한 줄로 적는다.

#### 6. 체크리스트

atomicity / secrets / tests + lint / SoT 일관 / commit 컨벤션 / PR 크기를 점검한다. PR 작성자가 실제 통과한 항목에 ☑를 달고, 사용자가 머지 전 final 검증으로 작성자 self-check의 false positive를 보정한다.

#### 7. 레퍼런스

plan / spec / 이전 PR / runbook / 메모리 링크.

### 2.2. 가독성 원칙

읽는 사람이 한 단계씩 따라 읽을 수 있도록 구조와 줄바꿈을 관리한다.

- **헤더 번호 매기기** — `1.1.1`처럼 점으로 구분하며 점 개수 = 깊이 − 1로 깊이를 직관적으로 표현한다.
  - `## 1. 개요` (섹션)
  - `### 1.1. 배경` / `### 1.3. 결과` (sub-섹션)
  - `#### 1.3.1. 해결한 문제` (세부 항목)
  - `### 2.1. {의사결정 단위}` (의사결정 단위)
  - `#### 2.1.1. 충돌 발생 과정` (의사결정 측면)
  - 필요 시 `##### 2.1.1.1`까지 사용.
- **헤더 깊이별 권장 sub-헤더 어휘** — 의사결정 측면을 분리할 때 명사형으로 통일한다.
  - `#### N.M.1 충돌 발생 과정`
  - `#### N.M.2 발견 과정`
  - `#### N.M.3 코드에 미치는 영향`
  - `#### N.M.4 처리 방법`
  - `#### N.M.5 다른 후보 처리`
- **소제목 문장 먼저, 명사형 불릿 나중** — 불릿 묶음 위에 소제목 성격의 한 문장으로 맥락을 제시한 뒤, 그 아래 불릿은 세부 내용을 명사형으로 적는다. 도입 문장 없이 불릿부터 나열하지 않는다.
  - **도입 문장도 명사형으로 끝낸다** — 불릿(또는 표)을 이끄는 도입 문장은 `~했다` / `~한다` 같은 서술형 종결을 쓰지 말고 명사형으로 끝낸다. 예: `전체 9주치 수치는 results/...에 있다.` → `전체 9주치 수치는 results/...에 기록.`
  - **본문은 산문 문단이 아니라 불릿** — 개요의 배경·목적·결과·기대 효과를 포함한 모든 섹션 본문은 줄글 문단이 아니라 명사형 불릿으로 적는다. 서술형 문단(`~한다.`로 이어지는 산문)은 쓰지 않는다.
- **줄바꿈 / 불릿** — 긴 문장에 `—` 또는 `→`가 여러 번 등장하면 한 줄에 묶지 말고 줄바꿈하거나 불릿으로 쪼갠다.
  - **한 줄 = 한 문장** — 한 불릿/문단에 마침표(`.`)로 끊기는 문장이 둘 이상이면 문장마다 줄(또는 불릿)을 나눠 한 줄에 한 문장만 둔다.
- **한국어 주석 강제** — PR 본문 안 코드 블록의 `#` 주석은 한국어로 단다.
- **표 셀** — 두 줄을 넘게 길어지는 셀은 셀 안에서 불릿(`-`)으로 쪼갠다.
- **범위 표기는 엔대시** — GitHub가 `~text~`를 취소선으로 렌더링하므로 숫자 범위에 `~`를 쓰지 않는다. `Task 1~12`처럼 `~`가 둘 이상 섞이면 그 사이가 취소선이 된다. 범위는 엔대시로 `Task 1–12` / `Phase 0–4`처럼 적는다.
- **코드 블록은 `--body-file`로 전달** — `gh pr create`/`edit`에 코드 블록(```` ``` ````)이 든 본문을 heredoc 인라인으로 넘기면 백틱·`$`가 escape되어 텍스트로 새어들어 펜스가 깨진다. 코드 블록이 있는 PR 본문은 파일에 작성해 `--body-file <file>`로 전달한다.

### 2.3. 어휘 원칙

- 영어 소제목 / 영어 narrative는 금지한다. 단, `type` / `scope` / 영문 식별자 / 코드 주석 / `Trade-off` 같이 정착한 외래어는 예외다.
- **면접 암시 표현 금지** — 커밋·PR·문서 본문에 '면접'·'면접관'·'인터뷰'·'인터뷰어' 등 면접을 암시하는 단어를 쓰지 않는다. 작업 동기·효과는 추적성·설명력·재현성·열람성 같은 중립 엔지니어링 표현으로 적는다.
- 두 단어 결합(의사결정 & Trade-off, 장애 시나리오 & 롤백 전략)은 `/`가 아닌 `&`로 통일한다.
- 시점 cutoff 원칙: 본 컨벤션 정착 *이전* PR(#1, #2)은 옛 11섹션 template 그대로 보존하고 retroactive 갱신을 하지 않는다. 컨벤션의 진화 자체가 포트폴리오의 자료다.

### 2.4. 크기 가이드

순수 코드 LOC 기준(자동 생성물 제외)으로 PR 크기를 관리한다.

- **≤ 200 lines** — 이상적(90% 확률로 1시간 내 review 완료, 결함 검출률 76%).
- **200~500** — 중간(분할 검토).
- **500+** — 큰 PR(분할 권장, 어려우면 description에 사유).

분할 단위는 다음을 권장한다.

- dbt의 "PR 1개 = 1 클래스 + 단위 테스트" 패턴.
- 본 프로젝트의 Task 단위. 이 저장소는 마일스톤 = Task로 잡아 PR = Task다(§2.5 참조).
- review fix는 그 fix가 발견된 본 PR 안에 포함한다(별도 PR로 빼지 않는다).

LOC 카운트에서 제외하는 항목은 다음과 같다.

- `uv.lock` / `package-lock.json` 등 자동 생성물.
- plan 본문 동기화 / docs 변경(review 부담이 적음).

출처: [optimal-pull-request-size](https://smallbusinessprogramming.com/optimal-pull-request-size/)

### 2.5. Merge 정책

**Squash merge**를 사용한다. main에는 마일스톤 단위로 1 commit만 들어간다. squash 전 개별 commit과 리뷰 코멘트는 PR 페이지에 영구 보존된다.

**이 프로젝트의 PR 해상도(마일스톤 = Task):** 결정·트러블슈팅을 PR 단위로 추적·설명하기 쉽도록, 이 저장소에서는 마일스톤을 Task 단위로 잡는다. 즉 **PR = Task**다.

- 코드 또는 설계 결정이 있는 Task는 단독 PR로 연다(결정 1개 = PR 1개 → "의사결정 & Trade-off"가 그 결정에만 집중).
- 코드 변경이 없는 순수 검증 Task(예: 샘플 E2E 실행)는 그 코드를 검증하는 PR의 "검증" 섹션에 흡수한다(빈 diff PR 방지).
- TDD로 한 Task가 여러 commit(실패 테스트 → 구현 → 리뷰 fix)이 되어도 squash가 main에 Task당 1 commit으로 정리한다.
- 머지는 작성자(사용자)가 PR 셀프 리뷰·체크 후 직접 수행한다(§2.6).

### 2.6. 셀프 리뷰

1인 프로젝트라도 PR 생성 후 본인이 commit 단위로 리뷰 코멘트와 체크 표를 남긴다.

### 2.7. Claude 서명 금지

PR 본문에는 "Generated with Claude Code" 같은 Claude 서명을 넣지 않는다. AI 활용은 커밋 단위 트레일러(§1.3)와 README의 "AI 도구 활용" 섹션으로만 드러낸다.

## 3. 트러블슈팅 기록

트러블슈팅은 규모와 성격에 따라 기록 위치를 나눈다.

| 위치 | 어떤 트러블슈팅 | 분량 |
|---|---|---|
| Commit 본문 | 그 commit이 직접 해결한 이슈 | 5~15줄 |
| PR description "계획 대비 편차" | 그 PR의 plan 대비 차이 | 표 또는 불릿 |
| `docs/troubleshooting/<주제>.md` | 큰 이슈(의사결정이 얽힌 것) | 1~2 페이지 |
| `docs/portfolio/phase1a_v1.md` (Day 10) | 가장 임팩트 큰 3~5건 정제판 | 짧게 |

별도 트러블슈팅 문서가 가치 있는 기준은 다음과 같다.

- spec / docs와 reality의 gap.
- 의사결정 trade-off가 있는 fix.
- 운영 중 재발 가능성이 있는 이슈.

단순 오타나 1줄 fix는 별도 문서가 필요 없다.
